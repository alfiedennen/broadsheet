## Detail per bug

### BUG-008 · Markdown renderer — full marked + DOMPurify swap

- **Severity**: BLOCKER → FIXED
- **Phase / step**: F3 + F4
- **Area**: `packages/core/src/lib/blocks/renderers/MarkdownBlockRenderer.svelte`
- **What was wrong**: hand-rolled inline-only renderer with explicit
  doc-comment "Deliberately NOT a full markdown engine." Supported
  `**bold**`, `*italic*`, `` `code` ``, links — NOT `#` headings, NOT
  `_italic_`, NOT lists, NOT blockquotes. User-authored content with
  `# Heading` rendered as literal `#`.
- **Fix**: pipeline replaced with `interpolate(body)` → `marked.parse(...)` (GFM enabled) → `DOMPurify.sanitize(...)` with explicit allowlist. Added `uponSanitizeAttribute` hook to nuke `data:`, `javascript:`, `vbscript:`, `file:` URL schemes belt-and-braces.
- **Tests**: `tests/unit/markdown-renderer.spec.ts` — 23 passing tests covering: headings (h1-h6), emphasis (`**`/`__`/`*`/`_`/`~~`), lists, blockquotes, links, code blocks, tables, autolinks, XSS hardening (script tags / iframes / event handlers / dangerous URL schemes / data: URLs), URL allowlisting (http/https/relative/mailto/tel pass; data:/javascript:/vbscript:/file: blocked).
- **Bundle impact**: marked + DOMPurify add ~38KB gzipped. Acceptable trade-off for full markdown + security-vetted sanitization.

### BUG-003 · Phase G preset picker — Blank canvas defensive default

- **Severity**: BLOCKER → FIXED
- **Phase / step**: 3 + F2 (re-confirmed always-broken on factory-fresh)
- **Area**: `packages/core/src/lib/presets/index.ts` + `packages/core/src/routes/settings/pages/+page.svelte`
- **What was wrong**: all 4 original presets had `applicableWhen` predicates requiring persons or areas-with-climates. On any install where ALL 4 predicates fail (zero persons OR no climate-bearing areas), `applicablePresets()` returned `[]` and the `{#if presets.length > 0}` template gate suppressed the entire picker — including the previously-hard-coded Blank chip. Live diagnostic on the canary (instrumented probe build) confirmed the predicates DID return false in the empty-discovery boot window.
- **Fix**: added `blankCanvas` preset at index 0 of PRESETS with no `applicableWhen` — always applicable. Removed the redundant hard-coded `<button>Blank</button>` from the template; the `{#each presets}` iteration handles it now. Picker now renders ≥1 chip on every install.
- **Tests**: `tests/unit/presets.spec.ts` — 6 passing tests. Asserts blank is at index 0, has no predicate, builds a usable Hero-only page, and is the SOLE preset returned for empty discovery (so future maintainers can't accidentally add a too-restrictive predicate to it).
- **Verified**: live on factory-fresh canary — picker renders all 5 chips (Blank canvas, Person page, Wall tablet morning, Family status board, Energy at a glance).

### BUG-009 · mushroom-climate-card translator — heating dashboards now actually import

- **Severity**: SERIOUS → FIXED
- **Phase / step**: F3
- **Area**: `packages/core/src/lib/lovelace/translate.ts`
- **What was wrong**: `mushroom-climate-card` (the most popular HACS climate card) was UNSUPPORTED, dropping every TRV from imported pages. A user importing their "Heating" Lovelace view ended up with a page that had no climate controls.
- **Fix**: added `translateMushroomClimate` — emits an action-grid tile with stateBinding to the entity. Calls `homeassistant.update_entity` as the safe no-op tap action (real setpoint editing happens in HA's more-info dialog the user opens by tapping). Coverage = `partial` with note "Climate card → state-bound tile. Setpoint editing stays in HA."
- **Tests**: 2 added in `tests/unit/translate.spec.ts` covering the translator + the missing-entity fallback.
- **Verified**: re-imported the user's actual Wall Tablet → Heating view on the canary. Was: "6 climate cards UNSUPPORTED, dropped". Now: "6 PARTIAL state-bound tiles". The imported `/heating/` page now has heating controls in it.

### BUG-010 · Unsupported-card placeholder — no more silent drops

- **Severity**: SERIOUS → FIXED
- **Phase / step**: F3
- **Area**: `packages/core/src/lib/lovelace/translate.ts` (visit fallback)
- **What was wrong**: ANY card type without a translator was silently dropped from imported pages. Users had no way to know cards had been there.
- **Fix**: visit fallback now emits a markdown block: `> _Unsupported \`<type>\` for \`<entity_id>\` — open in HA._` (or just `> _Unsupported \`<type>\` — no broadsheet translator yet._` if no entity field). Coverage stays `unsupported` so the report counter is honest, but the user SEES that something was there.
- **Tests**: 2 added in `tests/unit/translate.spec.ts` — asserts placeholder emits, entity_id embedded when available.
- **Verified**: Wall Tablet import skipped count went from 15 → 2 (87% reduction). Partial count rose from 87 → 100 to absorb the previously-skipped cards as placeholder blocks. Net 13 cards rescued from invisibility.

### BUG-005 · Slug humanizer — no more `alfies_office` in hero italic

- **Severity**: SERIOUS → FIXED
- **Phase / step**: F1
- **Area**: `packages/core/src/lib/utils/humanize.ts` (NEW) + `packages/core/src/lib/discovery/domain.ts`
- **What was wrong**: HA area names like `alfies_office`, `Utility_Room`, `library` (lowercase) leaked into rendered hero italic on `/lights`, `/heat`, `/wall` when no curation rename existed. Read like the dashboard was broken.
- **Fix**: `humanizeAreaName(raw)` heuristic — if the name looks like a slug (has underscore OR is all-lowercase single word), title-case + space. User-shaped multi-word names (`Front Hallway`, `Bedroom`) pass through. Wired into `domain.ts` so `area.name = override?.rename || humanizeAreaName(a.name)` — curation rename still wins, humanizer is the fallback.
- **Tests**: 5 cases in `tests/integration/fresh-curation.spec.ts` covering all the failure shapes from the dogfood.
- **Verified**: live `/lights` hero now reads "Alfies Office, Bedroom, Kitchen, and Living Room are on" (was "alfies_office, Bedroom, Kitchen, and Living Room are on").

### BUG-007 · PresenceCards procedural fallback

- **Severity**: SERIOUS → FIXED
- **Phase / step**: F1
- **Area**: `packages/core/src/lib/components/PresenceCards.svelte`

### BUG-011 · Block picker filter + scrollable container + count

### BUG-012 · Editor helper updated to point at HA-style states() + diagnostic note

- **Severity**: SERIOUS → FIXED (educationally)
- **Area**: `packages/core/src/routes/settings/pages/[slug]/+page.svelte` (markdown block editor hint)
- **What was wrong**: user wrote `{{(sensor.x.state | float * 100) | round(0)}}` and got literal "NaN" with no error. Cause: the legacy `{{entity.id}}` shorthand is direct-lookup-only and doesn't combine with filters; the user needed `{{ states('sensor.x') | float * 100 | round(0) }}`.
- **Fix**: editor hint now explicitly shows the working syntax + adds "output that reads NaN/undefined usually means a typo'd entity_id or a filter on missing data". Underlying Jinja was always correct.

### BUG-013 · Weather attribute access via legacy shorthand

- **Severity**: MINOR → DEFERRED to v0.1.1
- **Area**: legacy `{{entity.id}}` shorthand in markdown
- **Why deferred**: same root cause as BUG-012. The shorthand is direct state lookup — it doesn't traverse `.attributes.temperature`. Users who need attributes should write `{{ state_attr('weather.x', 'temperature') }}`. Documenting + fixing the underlying resolver to support nested attribute access is v0.2 territory.
